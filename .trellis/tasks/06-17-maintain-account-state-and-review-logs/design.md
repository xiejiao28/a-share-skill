# 账户状态记录与手动复盘设计

## 范围

本任务只负责：

- 建立账户状态、操作记录、复盘记录的目录结构
- 支持用户手动输入或更新账户信息与持仓状态
- 支持用户手动记录操作内容与原因
- 支持用户手动触发复盘
- 在复盘时对比上一次复盘之后的数据变化
- 在每次复盘后输出建议并落盘

不负责：

- 自动定时复盘
- 实盘券商接口接入
- 自动替用户生成操作原因

## 设计目标

在现有仓库能力上，补一个可持续维护的个人账户日志工作流，使智能体在后续会话里可以：

1. 读取最近一次账户状态
2. 读取最近一次复盘
3. 读取两次复盘之间新增的操作记录
4. 结合仓库里的账户查询/策略分析流程，生成新的复盘建议
5. 将新的账户快照、操作记录或复盘记录写回仓库

目标不是搭完整交易系统，而是让账户维护与复盘形成稳定的文件化流程。

## 现有能力约束

当前仓库已具备三类可复用基础：

### 1. 账户与交易查询

`a-share-paper-trading` 已能提供：

- `show-account`
- `positions`
- `orders`
- `trades`
- `run-snapshots`

以及引擎侧：

- `snapshot_accounts()`
- `account_snapshots`

### 2. 实验结果与报告落盘模式

`experiments/results/` 与 `experiments/reports/` 已经建立了：

- JSON 结果落盘
- Markdown 报告落盘

这说明仓库当前接受“脚本生成结构化结果 + Markdown 摘要”的工作方式。

### 3. 用户手动触发

当前没有自动调度框架，因此本任务默认：

- 由用户手动触发“开始复盘”
- 由用户手动告知“这次操作了什么、为什么操作”

## 目录设计

在仓库根目录新增：

- `log/account/`
- `log/actions/`
- `log/reviews/`

职责划分：

### `log/account/`

保存账户与持仓状态快照。

建议每次记录一个 Markdown 文件，包含：

- 时间
- 账户名
- 总资产
- 可用现金
- 当前仓位
- 当前持仓列表
- 备注

### `log/actions/`

保存用户每次操作的记录。

建议每次记录一个 Markdown 文件，包含：

- 时间
- 操作类型（买入/卖出/加仓/减仓/清仓等）
- 股票代码与名称
- 数量 / 价格（如果有）
- 操作原因
- 关联账户

### `log/reviews/`

保存每次复盘记录。

建议每次记录一个 Markdown 文件，包含：

- 时间
- 复盘区间
- 相比上次复盘的账户变化
- 相比上次复盘的持仓变化
- 相比上次复盘新增的操作记录摘要
- 当前诊断
- 建议

## 文件命名设计

每条记录单独成文件，文件名带时间戳，便于顺序检索。

建议命名：

- `log/account/YYYY-MM-DD_HHMMSS_account.md`
- `log/actions/YYYY-MM-DD_HHMMSS_action.md`
- `log/reviews/YYYY-MM-DD_HHMMSS_review.md`

这样在脚本里只需按文件名排序，就能取到最近一条记录。

## 复盘流程设计

### 输入来源

复盘时可综合三类输入：

1. 最近一次账户状态记录
2. 最近一次复盘记录
3. 自上次复盘后新增的操作记录

如果可用，也可以补充调用仓库现有工具：

- 查询账户
- 查询持仓
- 查询订单
- 查询成交
- 查询账户快照

### 复盘核心逻辑

每次“开始复盘”时：

1. 找到最新一条复盘记录
2. 读取该时间之后的所有 `log/actions/` 记录
3. 找到最近的账户状态记录
4. 如果有模拟盘账户，则补充读取当前账户/持仓/订单/成交
5. 形成变化摘要：
   - 账户净值变化
   - 仓位变化
   - 持仓变化
   - 新增操作及其原因
6. 输出新的复盘结论和建议
7. 写入新的 `log/reviews/` 记录

## 设计原则

### 1. 文件优先，方便跨会话持续使用

所有记录必须落盘，而不是只保留在对话中。

### 2. 结构简单，可手动维护

格式应允许：

- 用户手工补录
- 智能体自动追加

不要依赖复杂数据库或额外服务。

### 3. 复盘基于“上次复盘以来的变化”

本任务的关键不是单次静态分析，而是让每次复盘都具备时间连续性。

### 4. 建议必须写入复盘记录

这样下次复盘时，智能体可以对照：

- 上次建议是什么
- 这次实际做了什么

## 风险与边界

### 风险 1：用户输入信息不完整

如果账户状态或操作原因缺失，复盘质量会下降。

应对：

- 记录模板要尽量明确字段
- 缺失时允许保留空项，但不要伪造

### 风险 2：日志分散后检索困难

应对：

- 用固定子目录
- 用时间戳文件名
- 后续脚本统一按时间排序读取

### 风险 3：真实账户与模拟盘数据混用

应对：

- 记录中明确账户来源
- 如果是手工输入的真实账户状态，要标记为手工维护

## 后续扩展

第一阶段完成后，可以继续扩展到：

1. 自动生成记录模板
2. 自动从模拟盘账户抓取状态并写入 `log/account/`
3. 复盘中自动引用策略分析结果
4. 对比“上次建议”与“本次实际操作”的偏差
