# 记录账户状态与手动复盘流程

## 目标

在现有仓库框架上，补一套可持续维护的账户状态记录与手动复盘流程，使用户可以：

- 输入或更新自己的账户信息与持仓状态
- 在每次操作后记录操作和原因
- 在手动触发“开始复盘”时，让智能体调用仓库里的工具与流程完成复盘
- 把每次复盘结论和建议记录下来
- 在下一次复盘时，自动比较两次复盘之间的数据变化

## 已知信息

* 用户希望新建一个目录：
  * `log/`
* `log/` 目录下每次记录都新建一个 Markdown 文件，并标记时间。
* 用户希望维护的信息包括：
  * 账户信息
  * 持仓状态
  * 操作记录
  * 复盘记录
* 用户会手动告诉智能体每次操作及其原因。
* 用户会手动触发复盘，例如说“开始复盘”。
* 当前仓库里已经存在可复用能力：
  * `a-share-paper-trading` 可查询账户、持仓、订单、成交
  * `a-share-paper-trading` 已支持账户快照 `snapshot_accounts`
  * `experiments/results/` 与 `experiments/reports/` 已形成结果/报告落盘模式

## 临时假设

* 本任务优先复用现有查询能力和报告落盘模式，不重写交易引擎。
* 记录目录会放在仓库内，而不是外部路径。
* 复盘以“用户手动触发”为前提，不做自动定时调度。

## 需求

* 新增 `log/` 目录用于账户状态、操作与复盘记录。
* `log/` 下采用分子目录结构：
  * `log/account/`
  * `log/actions/`
  * `log/reviews/`
* 每次新增记录都生成一个带时间标记的 Markdown 文件。
* 支持维护或更新：
  * 账户信息
  * 当前持仓
  * 操作记录
* 支持一次手动触发的复盘流程：
  * 读取当前账户/持仓状态
  * 对比上一次复盘之后的数据变化
  * 输出复盘建议
  * 写入复盘记录
* 每次用户执行操作后，支持记录：
  * 操作内容
  * 操作原因
* 每次复盘后必须给出建议。

## 验收标准

* [ ] 已新增 `log/` 目录
* [ ] 已定义账户状态 / 操作 / 复盘记录的落盘方式
* [ ] 已支持手动触发复盘
* [ ] 复盘会比较两次复盘之间的数据变化
* [ ] 每次复盘会输出建议并记录
* [ ] 每次操作可记录操作和原因

## 明确不做

* 自动定时复盘
* 实盘券商接口接入
* 自动替用户生成操作原因

## 开放问题

* 当前无阻塞性开放问题，规划已足够继续细化设计。
