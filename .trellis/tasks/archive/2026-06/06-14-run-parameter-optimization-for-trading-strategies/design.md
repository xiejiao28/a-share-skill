# 交易策略参数调优设计

## 范围

本任务只设计并实现“单票参数调优”的第一阶段能力，目标对象为：

- `sma_cross`
- `rsi_revert`

不纳入参数调优范围：

- `buy_and_hold`
  - 仅作为基准回测对照

单票验证标的固定为：

- `601669`

默认回测区间：

- 近 3 年

## 设计目标

在不重写现有回测引擎的前提下，为当前仓库补上一层“实验编排”能力，使参数搜索可以：

1. 复用 `a-share-paper-trading` 的现有回测逻辑
2. 把实验脚本放到 `experiments/scripts/`
3. 把原始结果落到 `experiments/results/`
4. 后续可以自然扩展到股票池批量验证

## 现有能力约束

### 现有回测入口

当前仓库已有统一回测入口：

- `a-share-paper-trading/scripts/paper_trading/engine.py`
  - `PaperTradingEngine.run_backtest(...)`

这个接口已经支持：

- `buy_and_hold`
- `sma_cross`
- `rsi_revert`

并且对参数的支持已经内置：

- `sma_cross`
  - `fast`
  - `slow`
- `rsi_revert`
  - `buy_rsi`
  - `sell_rsi`

### 现有输出特征

当前 `run_backtest(...)` 已能产生：

- 交易记录
- 权益曲线
- 最终净值
- 最大回撤所需基础数据

因此第一阶段不应该重做撮合引擎，而应该在现有回测结果之上增加“批量试参 + 排序”。

## 设计方案

### 方案核心

新增实验编排脚本，负责：

1. 读取固定实验输入
   - 股票代码：`601669`
   - 时间区间：近 3 年
   - 初始资金：固定默认值，后续可参数化
2. 枚举参数组合
3. 调用现有回测引擎逐组回测
4. 汇总结果并计算排序指标
5. 输出机器可读结果文件

### 目录落点

- `experiments/scripts/`
  - 放实验脚本
- `experiments/results/`
  - 放 JSON / CSV 结果
- `experiments/reports/`
  - 后续人工或脚本生成实验结论

### 脚本边界

实验脚本是“编排层”，不直接复制回测算法。

职责：

- 组装参数网格
- 调用回测
- 汇总排序
- 写结果文件

不负责：

- 改写交易规则
- 改写撮合逻辑
- 生成自然语言结论

## 子任务拆分

### 子任务 1：`sma_cross`

目标：

- 搜索 `fast` / `slow` 参数组合

基本约束建议：

- `fast < slow`
- `fast` 不宜过大
- `slow` 需要覆盖中期趋势

第一阶段重点：

- 先验证脚本框架和排序逻辑
- 首版固定使用中等网格：
  - `fast`: `3, 5, 8, 10, 12, 15`
  - `slow`: `20, 30, 40, 60`

### 子任务 2：`rsi_revert`

目标：

- 搜索 `buy_rsi` / `sell_rsi` 参数组合

基本约束建议：

- `buy_rsi < sell_rsi`
- 两者保留足够阈值间距，避免频繁翻转

第一阶段重点：

- 与 `sma_cross` 对比风格差异，而不是只比总收益

## 评价指标设计

主原则：

- 收益 / 回撤平衡

因此第一阶段排序不应只看总收益，而应至少同时输出：

- 总收益率
- 最大回撤
- 收益回撤比
- 交易次数

建议默认排序逻辑：

1. 先过滤明显不可用结果
   - 无交易或交易极少
   - 回撤异常高
2. 再按收益回撤比优先
3. 同比下参考总收益率

## 输出契约

第一阶段建议输出统一 JSON 结构，至少包含：

- `strategy`
- `symbol`
- `start`
- `end`
- `params`
- `final_equity`
- `return_pct`
- `max_drawdown`
- `return_drawdown_ratio`
- `trade_count`
- `rank`

后续如需做表格分析，再从 JSON 衍生 CSV。

## 风险与边界

### 风险 1：单票过拟合

只对 `601669` 做调优，结果只代表这只票在近 3 年的局部表现，不代表策略泛化能力。

应对：

- 在 PRD 和结果文件中明确标记“单票验证”

### 风险 2：收益高但不可交易

如果只看收益，容易选出回撤过大或交易频率不合理的参数。

应对：

- 把回撤约束放进默认排序逻辑

### 风险 3：编排层和引擎层重复实现

如果实验脚本自己实现信号逻辑，会和正式回测引擎漂移。

应对：

- 实验脚本必须调用 `run_backtest(...)`，不复制策略实现
